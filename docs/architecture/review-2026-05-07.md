## 1. What's actually good

Worth saying explicitly so the criticism below is calibrated:

- **Real DDD layering, mechanically enforced.** `scripts/tests/test_ddd_architecture_guardrails.py` is a proper guardrail: AST-walks every `domain/`, `application/`, `router*` file and asserts no FastAPI in domain, no DAL in routers, no `run_sql_async`/`tbl` in routers, no unauthorized bounded contexts. It even has a meta-test (`test_router_guardrail_detects_nested_router_dal_import`) that proves the linter itself catches violations. This is a level of architectural discipline most repos don't have.
- **Real shared libraries.** `libs/shared-{api,auth,db,domain,frontend-api,frontend-i18n,ui,trace,geo}` are workspace-managed (uv on Python, npm workspaces on TS). `shared-api/app_factory.py` centralises the FastAPI middleware stack so every app gets identical CORS/rate-limit/same-origin/latency behaviour.
- **Parameterised SQL only.** Verified — no f-string SQL on user-supplied values; `sql_param()` and `:name` parameters are used consistently. `tbl()` is the *one* documented dynamic-SQL exception, and the catalog/schema come from env, not requests.
- **`safe_global_exception_response`** correctly returns generic 500s with an `error_id` and logs the stack trace server-side only. No traceback leakage to clients.
- **Multi-language i18n enforced** — `scripts/validate_i18n.py` is a pre-commit hook, all 14 locales kept in sync. This is rare and hard to maintain, and you've kept it.
- **ai-context as a contract.** `entities.yaml`, `joins.yaml`, `metrics.yaml`, the working rules — this is the right pattern for an AI-assisted codebase. It works as a guardrail against hallucinated tables and spurious joins.

## 2. CRITICAL issues

### C1. The platform shell duplicates every app backend, byte-for-byte, with no automation in the repo

```
diff -rq apps/platform/processorderhistory_backend apps/processorderhistory/backend/processorderhistory_backend
# (returns nothing — they're identical)
```

`apps/platform/processorderhistory_backend/`, `warehouse360_backend/`, `connectedquality_backend/` are *exact copies* (168 vs 127 .py files because platform also has `shared_*` and `backend/`). The `deploy.toml` says the build step is `python3 scripts/build.py` — **that file doesn't exist** at `apps/platform/scripts/`. It exists only in `.claude/worktrees/`, i.e. inside untracked agent sandboxes.

Implications:
- **How is the platform bundle currently being kept in sync?** Either manually copy-pasted (drift inevitable) or by an unchecked `build.py` that lives only in a developer's worktree (not reproducible).
- The `router_imwm` bug we just fixed is a direct consequence: someone added the entry to `apps/platform/backend/main.py` but the source `router_imwm.py` was never copied across because the build is broken.
- **Severity is critical because every UAT incident now starts with "is platform out of sync with `apps/<x>/backend/`?"**

**Fix:** Either
1. Restore `apps/platform/scripts/build.py` (committed, with tests), or
2. Replace the duplication with `pip install -e ../<app>/backend` per app in `apps/platform/requirements.txt` and import the *real* package — the platform is then a thin orchestrator, not a copy.

Option 2 is the correct long-term fix. Option 1 is the fastest fix to stop the bleeding.

### C2. `apps/platform/databricks.yml` `sync.include` references the old names

```yaml
sync:
  include:
    - app.yaml
    - backend/**
    - static/**
    - poh_backend/**          # ← does not exist
    - cq_backend/**           # ← does not exist
    - shared_db/**
    - shared_api/**
    - shared_auth/**
```

Real directories are `processorderhistory_backend`, `connectedquality_backend`, `warehouse360_backend`, plus `shared_domain`. The deploy worked because `databricks bundle deploy` defaults to including everything in the bundle root when `source_code_path: .`, but the `sync` block is now *worse than nothing*: it's documentation-by-config that's actively misleading.

**Fix:** Either delete the `sync.include` block (let the bundle default scope it) or update it to current directory names plus `processorderhistory_backend`, `warehouse360_backend`, `connectedquality_backend`, `shared_domain`.

### C3. `_optional_router` silent failure is too quiet for production

```python
def _optional_attr(module_name, attr_name, artifact):
    try:
        return getattr(import_module(module_name), attr_name)
    except Exception as exc:
        _missing_build_artifacts[artifact] = error_str
        logger.warning(...)
        return None
```

When `router_imwm` failed to import, the platform happily started up, served all OTHER routes, and only flipped `/api/ready → 503`. **Every route from the broken artifact silently returns 405/404** (depending on whether home StaticFiles is mounted). There is no metric, no alert, no startup failure. You only find out from a customer ticket.

Worse: the same artifact key (`"warehouse360_backend"`) is shared across all eight W360 routers, so a single broken file in one router clobbers the readiness state for the whole package even when the other seven loaded fine.

**Fix:**
- Distinguish *expected-optional* artifacts (CQ/POH/W360 absent in a debug build) from *required-now* artifacts. Anything in `app.template.yaml`'s deployment scope should *fail startup*, not warn.
- Per-router artifact keys, not one shared `"warehouse360_backend"` bucket.
- Push the `_missing_build_artifacts` dict to logs at ERROR level if any expected artifact is missing, and add a `/health/routers` endpoint that lists registered route count by tag.

### C4. `AUTH_ALLOW_UNVERIFIED_JWT=true` hardcoded in `app.template.yaml`

`apps/platform/app.template.yaml:55-56` — value is *literal* "true", not a `${...}` substitution. The `deploy.toml`'s `AUTH_ALLOW_UNVERIFIED_JWT=false` is *ignored* because there's no template variable to substitute into. Comment says: *"Always true in this deployment context — the proxy handles verification upstream."*

That's true *today, in Databricks Apps*. It is *not* defence-in-depth — it is single-line-of-defence. The threat model is:
- Direct port access to the container (e.g., misconfigured network policy, port-forward in dev with a real token).
- Migration to a different proxy that validates differently.
- Local debug session where someone copies a real JWT into curl.

**Fix:** Wire `AUTH_JWKS_URL` to the workspace IdP and flip `AUTH_ALLOW_UNVERIFIED_JWT` to false. The `warn_if_jwks_unconfigured()` boot warning at `shared_auth/identity.py:59-70` is the right hook — make it an error in non-dev environments.

### C5. CI declares 75% coverage but does not enforce it

`.github/workflows/ci.yml:51-59` uploads coverage artifacts. **Nowhere in the workflow does a step fail if coverage drops.** CLAUDE.md mandate vs reality:

```
3. **Test Coverage**: We enforce a minimum of 75% coverage across all
   applications in CI. A goal of 100% is encouraged...
```

This is enforced nowhere I can see. There is no `pytest-cov --cov-fail-under=75` flag, no `vitest --coverage --reporter` threshold, no codecov gate. Coverage drifts silently downward.

**Fix:** Add `--cov-fail-under=75` to the pytest invocation in each app's `nx test` target, and `coverage.thresholds.lines: 75` to `vitest.config.ts`. Surface this in PR status checks.

## 3. HIGH severity

### H1. Bearer-token fallback contradicts the proxy-only auth story

`libs/shared-auth/src/shared_auth/identity.py:32-56` — `resolve_token(strict=False)` falls back to `Authorization: Bearer ...`. The `require_user` dependency is `strict=False`. Most app routes use `require_proxy_user` (strict), but you should grep for `require_user` (non-strict) and decide per route.

If the only legitimate caller is the Databricks Apps proxy, **`strict=True` should be the default**, and non-strict should be opt-in for explicit dev-mode endpoints (and explicitly disabled when `APP_ENV` is not `development|local|test`).

### H2. SSRF surface in the Genie client

`apps/processorderhistory/backend/processorderhistory_backend/genie_assist/application/genie_client.py:48-63`:

```python
host = os.environ.get("DATABRICKS_HOST") or os.environ.get("DATABRICKS_HOSTNAME") or ""
if not host.startswith(("http://", "https://")):
    host = f"https://{host}"
return host
```

`urllib.request.urlopen` then POSTs to `{_host()}/api/2.0/genie/...` with the **user's bearer token**. If `DATABRICKS_HOST` is ever overridable from elsewhere (env file precedence, redeploy with a misconfigured target), every Genie call exfiltrates a Databricks token to the configured host. Validate against an allowlist on startup; fail closed if the host doesn't match `*.cloud.databricks.com` (or your azure/aws workspace pattern).

Same instinct: prefer the Databricks SDK over hand-rolled `urllib.request` for any new outbound call.

### H3. Rate limiter is per-process and IP-spoofable for unauthenticated paths

`libs/shared-api/src/shared_api/rate_limit.py:141-154` — falls back to first entry of `X-Forwarded-For` when no token. That header is client-controlled; an attacker rotates it to bypass per-IP limits.

The 10K-bucket cap (`max_buckets`) means an attacker with rotating XFF values can also evict legitimate users' buckets and then spam them. In a multi-replica deployment each pod has its own state — limits don't compose.

This is *acceptable for a single-pod Databricks App with proxy-validated tokens*, where every real request has a token and the per-token bucket is the only one that matters. Document that explicitly so it's not assumed to be a general-purpose limiter, and make sure no public/unauthenticated endpoints rely on it.

### H4. No security scanning in CI

`.github/workflows/ci.yml` runs lint/typecheck/test/build. No:
- `pip-audit` / `safety` for Python deps.
- `npm audit` (with a fail threshold) for JS deps.
- Static analysis (`bandit`, `semgrep`, CodeQL).
- Container/image scanning (you don't build images, but you should scan the resulting bundle if there are wheels with known CVEs).
- Secret scanning (GitHub native or `gitleaks`).

For a repo that handles a Databricks token via `x-forwarded-access-token` and runs SQL against gold data, this is below table stakes.

**Fix:** Add a `security` job parallel to `ci` that runs `pip-audit`, `npm audit --audit-level=high`, and CodeQL (free for public repos / available via Advanced Security for private). Treat as required check.

### H5. No fine-grained authorization layer

Every `Depends(require_proxy_user)` confirms identity but never *authorizes*. The user gets back any plant's data so long as Unity Catalog allows the underlying token to read the gold view. That works *today* because UC enforces row/table policies, but:
- You have no audit trail of "user X requested plant Y data they shouldn't have" — UC denials surface as opaque DB errors.
- Adding any feature that aggregates *across* plants (e.g., a portfolio dashboard) implicitly trusts UC permissions to be perfect.
- `PlantScope.from_optional(plant_id)` accepts whatever the request says with no check that the user belongs to that plant.

`libs/shared-api/errors.py` even has an `UnauthorizedPlantAccess` class — *and it's never raised anywhere*. It's a stub waiting for the layer that should call it.

**Fix:** Read groups from JWT, materialise an `AuthorizationContext` in the request, and have routers/application services check plant/group membership before delegating to DAL.

### H6. CLAUDE.md is stale

> "This is an Nx monorepo containing core Databricks Apps:
> envmon, spc, trace2, warehouse360, poh"

Reality: `apps/` has *seven* directories — also `connectedquality` and `platform`. Agents (and humans) using this file as orientation get a wrong mental model from line 1. Keep this file as the canonical "open this first" doc — make it correct. Add `connectedquality` and `platform` rows; describe the platform shell explicitly.

## 4. MEDIUM severity (architectural smells)

### M1. The `apps/platform/` bundle is doing two jobs at once

It's both **the deployment unit** *and* **a gateway/composite shell over five apps**. Those two responsibilities have different change cadences:
- The deployment unit changes when you tweak Uvicorn/Databricks bundle config.
- The shell composition changes whenever any of the five apps changes.

Today they share `main.py` and one routers list. A cleaner split:
- `apps/platform/` becomes the deploy unit.
- The composition (CQ_ROUTERS, POH_ROUTERS, W360_ROUTERS) lives in `libs/shared-platform-router-registry/` (or in each app's own `__init__.py` exposing a `register(app, prefix)` function).
- Each app self-registers via a manifest the platform discovers, instead of platform listing every router by import path.

This converts `_optional_router` from "silently swallow ImportError" to "if a registered app failed to provide a register hook, fail loud."

### M2. The `_include_available_routers` loop in `apps/platform/backend/main.py` is the wrong abstraction

```python
for router, prefix, tags in [*CQ_ROUTERS, *POH_ROUTERS, *W360_ROUTERS]:
    if router is None: continue
    kwargs = {"prefix": prefix}
    if tags is not None: kwargs["tags"] = tags
    app.include_router(router, **kwargs)
```

`tags is not None` is the only branch logic; tag fan-out is per-router, but the `prefix` is identical inside each app group (`/api/cq`, `/api/wh`, `/api`). The code reads as "we knew exactly what we wanted but we're pretending it's data-driven."

A cleaner shape: each app exposes a list of `(router, sub_prefix, tags)` and the platform mounts each app's whole list under its base prefix. Then `W360_ROUTERS = wh360.compose_routers()` and the platform never imports individual modules.

### M3. The `_missing_build_artifacts` global is a soft singleton

`apps/platform/backend/utils.py:7` — module-level mutable dict. Imports during module load mutate it. Tests can't reset it without reaching into module internals. No locking. It's fine because it's only mutated at startup, but it's fragile under module-reload tools (pytest with importlib reload, hot-reload uvicorn). Prefer returning a result object from a setup function.

### M4. SQLite preferences DB at `/tmp/connectio_platform_prefs.db`

Per the security agent's finding — ephemeral tmpfs in a container, lost on restart, distinct per pod replica. For *user UI preferences* (pinned modules, density), this is *acceptable but should be visible*: the UI should treat preference-load 404 as "first time, here are defaults" rather than as data loss.

If you ever want preference durability (and Databricks Apps does have persistent storage options), this is a 2-day move to a small Unity Catalog managed table. Don't put it on the roadmap *unless* you've heard this is hurting users — it's the right level of effort for the value.

### M5. `request.completed` logged at INFO on every request

`libs/shared-api/middleware.py:65-70` — at moderate scale this drowns the log channel. Drop to DEBUG; INFO is already used at line 72 for budget breaches, which is the right signal.

### M6. The `[email protected]` permission model is wide-open

Comment in `databricks bundle deploy` output: *"the bundle root path /Workspace/Shared/.bundle/platform/uat is writable by all workspace users."* That's a Databricks default the bundle didn't override. For UAT it's probably fine; for prod it lets any workspace user with shared-folder rights modify the bundle source code.

**Fix:** in `databricks.yml`, scope `workspace.root_path` to a service-principal-owned folder, and add explicit `permissions:` entries.

### M7. Pre-commit hooks list per-app architecture tests by full path

`.pre-commit-config.yaml:11-19` enumerates `apps/envmon/backend/tests/test_architecture_boundaries.py`, etc. Adding a new app means editing the hook config. Use a glob (`apps/*/backend/tests/test_architecture_boundaries.py`) or invoke the central guardrail at `scripts/tests/test_ddd_architecture_guardrails.py` (which is already comprehensive and checks all six apps).

### M8. The pre-commit hook does **not** run the central DDD test

`scripts/tests/test_ddd_architecture_guardrails.py` is the more comprehensive check (router/application/domain rules + bounded context allowlist). The pre-commit only runs the per-app `test_architecture_boundaries.py` files. So adding a new bounded context only fails on push (CI) rather than at commit time.

### M9. `CLAUDE.md` and the working rules duplicate the "10/10 docs" mandate but the codebase doesn't meet it uniformly

Sample: `apps/platform/backend/routes/session.py` — has the PEP 257 docstring (we just fixed it). `apps/processorderhistory/backend/.../orders_dal.py:44-55` — same standard. **But also**: untracked work in progress (`apps/warehouse360/backend/inventory_management/`) is sitting on the file system not yet bundled, and any agent picking it up has no way to know it exists. The mandate is good; the enforcement is sporadic.

There is no automated check (e.g., `interrogate` or `flake8-docstrings`) that verifies docstring presence. Add it; trust the tooling, not human discipline.

### M10. Two FastAPI app-creation paths — `create_api_app()` and `register_spa_routes()`

Standalone apps use `create_api_app()` + `register_spa_routes()` (route-based SPA serving with safe path traversal checks). The platform uses `create_api_app()` + raw `app.mount("/", StaticFiles(html=True))`. **Different security profile**: `safe_static_file()` in `register_spa_routes` resolves and bounds the path to root via `is_relative_to`, while `StaticFiles` relies on Starlette's resolution. They likely both work, but the divergence means a fix in one doesn't apply to the other.

**Fix:** Have the platform use `register_spa_routes()` for the home shell. Keep mounted SPAs at `/cq`, `/poh`, etc. as `StaticFiles` (they're per-prefix and don't catch-all the API namespace).

## 8. Implementation status — CRITICAL findings (C1–C5)

All five CRITICAL findings have been addressed on `feature/pr2-arch-guardrails` in a single coherent change set. Local verification details below; deployment to UAT remains gated on user approval.

### C1 ✅ — Platform duplication replaced with built wheels

- `apps/platform/scripts/build.py` rewritten to call `uv build` for each platform-required package and emit wheels into `apps/platform/wheels/`. Twelve wheels produced per build: 6 shared libs (`shared-api`, `shared-auth`, `shared-db`, `shared-domain`, `shared-trace`, `shared-geo`) and 6 app backends (`connectedquality`, `envmon`, `processorderhistory`, `spc`, `trace2`, `warehouse360`).
- `apps/platform/requirements.txt` references each wheel by exact filename (e.g. `./wheels/shared_api-0.1.0-py3-none-any.whl`) so pip never falls through to PyPI for any internal package name. (We discovered the hard way that `shared-db` exists on PyPI and `--find-links` alone wasn't sufficient.)
- `apps/platform/.gitignore` updated: `wheels/` is gitignored build output; the old per-package source-copy entries (`poh_backend/`, `cq_backend/`, `connectedquality_backend/`, `processorderhistory_backend/`, `warehouse360_backend/`, `shared_api/`, `shared_auth/`, `shared_db/`, `shared_domain/`) removed because the new build no longer creates them.
- `apps/platform/Makefile` gains a `dev-install` target that does editable installs from each upstream package — local source edits are picked up immediately without rebuilding wheels.
- **Verification:** fresh venv + `pip install -r requirements.txt` (run from `apps/platform/`) installs all internal packages from local wheels. `python -c "import backend.main"` produces **64 routes registered, zero `Optional artifact unavailable` warnings, zero entries in `_missing_optional_artifacts`** (vs. the pre-fix state of 27 routes plus 28 `shared_db.core` not-found warnings).

### C2 ✅ — `databricks.yml` sync block fixed

- Stale `poh_backend/**`, `cq_backend/**` includes removed.
- `wheels/**` and `requirements.txt` added (these are what runtime needs now).
- `sync.exclude` block added for `__pycache__`, `*.pyc`, `.pytest_cache`, `.mypy_cache`, `.ruff_cache`, `node_modules`, plus dev-only files (`scripts/`, `deploy.toml`, `Makefile`, `app.template.yaml`, `frontend/`, `standalone/`).
- **Verification:** `databricks bundle validate --target uat` passes. The four warnings emitted are pre-existing (the `/Workspace/Shared` writability warning) or cosmetic (unused exclude patterns matching no current files).

### C3 ✅ — Required vs optional artifact classification

- `apps/platform/backend/utils.py` gains a `required: bool` parameter on `_optional_attr`. When `required=True` an import failure raises `RequiredArtifactMissing` (a new exception class) — *the platform refuses to start* rather than silently 404-ing the affected routes. Optional failures are recorded in `_missing_optional_artifacts` keyed by full module path (no more shared `"warehouse360_backend"` bucket clobbering unrelated entries). The legacy `get_missing_artifacts()` is preserved as an alias.
- `apps/platform/backend/main.py` rewritten to call a new `_required_router(module_name)` helper for every router currently in `CQ_ROUTERS`, `POH_ROUTERS`, `W360_ROUTERS`. The `_required_attr` helper does the same for the two POH module-level imports (`check_warehouse_config`, `run_sql_async`).
- `_include_available_routers` renamed to `_include_required_routers` — the None-check is gone because the imports are now load-bearing.
- New endpoint `GET /api/health/routers` returns the registered route inventory and any optional-artifact failures, so ops can see drift at a glance.
- `/api/ready` simplified — required failures abort startup, so the runtime probe only needs to check warehouse connectivity.
- **Verification:** new tests `apps/platform/backend/tests/test_utils.py` and updated `test_main.py` cover the required/optional split, the health endpoint, and the per-module artifact key. `RequiredArtifactMissing` raises with a clear message when a required module is unavailable (verified via a monkeypatch test). All 11 platform backend tests pass.

### C4 ✅ — JWT verification wired (UAT) and enforced in non-dev

- `apps/platform/app.template.yaml`: hardcoded `AUTH_ALLOW_UNVERIFIED_JWT: "true"` replaced with `${AUTH_ALLOW_UNVERIFIED_JWT}`. New `AUTH_JWKS_URL` and `AUTH_JWT_ISSUER` template variables added. Comment rewritten to describe defence-in-depth instead of "always true."

### Post-deploy follow-up — W360 schema mismatch (commits `761012d`, `b783063`)

After the C1–C5 deploy proved the routing layer was clean, two W360 endpoints surfaced **500 errors** that had been masked behind the previous 404s:

```

`connected_plant_uat`.`gold`.`wh360_process_orders_v` cannot be found.
```

Root cause: `shared_db.core.tbl()` resolves all queries to `<TRACE_CATALOG>.<TRACE_SCHEMA>.<view>`, and the platform shell sets `TRACE_SCHEMA=gold` because CQ and POH live there. But the warehouse360 SQL views are deployed under `<catalog>.wh360.*` (per `apps/warehouse360/sql/views/01_wh360_process_orders_v.sql`). Standalone W360 worked because its own `deploy.toml` sets `TRACE_SCHEMA=wh360`; the platform-bundled W360 did not.

**Fix:** `warehouse360_backend/utils/db.py` now defines a W360-specific `tbl()` that reads a new `WH360_SCHEMA` env var (falling back to `TRACE_SCHEMA`, then to `gold`). The platform's `app.template.yaml` and `deploy.toml` set `WH360_SCHEMA=wh360` for both UAT and prod; the standalone W360 app keeps using the `TRACE_SCHEMA` fallback. The `DataFreshnessRuntime` was retargeted to `WH360_SCHEMA` for consistency.

**One caveat surfaced during deploy:** the first redeploy after the schema fix still 500'd because the wheel version was unchanged at `0.1.0` and pip's same-name-same-version cache skipped reinstalling the new wheel. Bumping `warehouse360-backend` to `0.1.1` (and the matching filename in `requirements.txt`) forced a clean install. **Going forward, any wheel content change in this monorepo needs a version bump** — ratchet it whenever the source changes.
